Section 01

The Foundations You Need First

Before running a single command, let's make sure the mental model is solid. This section explains networks, MAC addresses, Ethernet frames, and broadcast — from scratch.

What is a computer network? At the most basic level, it's just computers connected so they can send messages to each other. That's it. Everything else — switches, routers, VLANs — is just machinery to make that message delivery work better.

What is Ethernet? Ethernet is the most common set of rules (protocol) that computers on a local network use to communicate. When computers on the same network talk to each other directly, they use Ethernet.

Analogy — The Post Office

Think of a network like a neighbourhood. Each house is a computer. The post office rules (Ethernet) say: every letter must have a sender address and a recipient address written on it. The delivery people (switches) use those addresses to get the letter to the right house.

MAC Addresses — Your Computer's Permanent Name Tag

Every single network card (physical or virtual) in the world has a MAC address (Media Access Control address). It looks like this: 06:e5:f0:20:af:7a. It's six groups of two hex digits, separated by colons.

Think of it as a serial number permanently stamped on the network card at the factory. Unlike an IP address (which you can change), a MAC address is burned in. When computers on the same local network want to find each other, they use MAC addresses.

MAC Address structure

f0

7a
Manufacturer ID (OUI)
Device Serial
Special address ff:ff:ff:ff:ff:ff = "send to EVERYONE" (broadcast)

Ethernet Frame — The Envelope

When computer A wants to send data to computer B, it doesn't just blast raw bytes. It wraps the data in an Ethernet frame — think of it as putting a letter in an envelope with proper labelling.

Structure of an Ethernet Frame
Destination MAC
6 bytes — Who gets this?
Source MAC
6 bytes — Who sent this?
EtherType
2 bytes — Type
Payload (Data)
46–1500 bytes — The actual message
Broadcast frame: Destination MAC = ff:ff:ff:ff:ff:ff → every device on the network must read it

What is a Broadcast?

Analogy — The Town Crier

A broadcast is when one device shouts to everyone at once: "Hey! Does anyone know where 192.168.1.5 is?" Every device on the network hears it, checks if the question applies to them, and either responds or ignores it. The special MAC address ff:ff:ff:ff:ff:ff means "deliver to all."

The set of all devices that receive this shout = broadcast domain.

The Big Question This Lab AnswersWhen you connect multiple network segments together, which devices can hear each other's broadcasts? The answer to that question defines your broadcast domain — and this whole lab is about understanding exactly that.
Section 02

Network Namespaces — Virtual Rooms

The lab uses "network namespaces" to simulate separate computers on a single Linux machine. Let's understand what that actually means.

Normally, your Linux computer has one network identity: one set of network interfaces, one routing table, one set of firewall rules. Everything on the machine shares it.

A network namespace is like drawing a partition line inside the operating system. Each namespace gets its own completely isolated network identity — its own virtual network cards, its own routing table, its own firewall. Devices in one namespace cannot see devices in another namespace unless you explicitly connect them.

Analogy — Separate Houses

Imagine one physical building (your computer) split into multiple separate apartments (namespaces). Each apartment has its own front door, own letterbox, own phone line. Tenants in Apartment A have no idea what Apartment B is doing — they can't shout through the walls. If you want Apartment A to talk to Apartment B, you must install a pipe between them (that's the veth pair, coming next).

One Linux machine, three isolated namespaces
All three "computers" are simulated on a single machine. They're invisible to each other until connected.

The ip netns command family

ip netns add host1
ip
The main Linux networking tool. Like a Swiss Army knife for network configuration. Almost everything network-related in modern Linux goes through ip.
netns
Subcommand: "network namespace". Tells ip that we want to work with namespaces.
add
Action: create a new namespace.
host1
The name we give this namespace. After this runs, a namespace called "host1" exists on the machine. Think of it as: "Build me a new isolated apartment called host1."
ip netns exec host1 <some command>
ip netns exec
"Go inside namespace and run a command." Without this, every command you type affects the default (root) namespace — your real machine. With exec host1, the command runs as if you're operating INSIDE the host1 apartment.
host1
Which namespace to enter.
Alternative — Use nsenter instead

The lab also uses nsenter — another way to "step inside" a namespace. It's more flexible:

nsenter vs ip netns exec
# Method 1: ip netns exec ip netns exec host1 ip link show # Method 2: nsenter (used in tcpdump commands later) nsenter --net=/var/run/netns/host1 ip link show # Both do the same thing: run 'ip link show' inside namespace host1 # nsenter uses the actual file path of the namespace # ip netns exec uses the namespace name directly
Why no IP addresses? This lab operates at Layer 2 (Ethernet) — the level of MAC addresses and frames. You don't need IP addresses to send Ethernet frames! Devices on the same local network can communicate using only MAC addresses. IP addresses (Layer 3) are only needed when crossing network boundaries via a router.
Section 03

Veth Pairs — The Virtual Cable

A veth (Virtual ETHernet) pair is like a physical network cable, except it's software. You can use it to connect two namespaces together.
Analogy — A Pipe Between Rooms

Remember our apartment analogy. A veth pair is like drilling a hole through the wall between Apartment A and Apartment B and inserting a pipe. Anything you push into one end comes out the other end.

A veth always comes in pairs — you can't have one end without the other. One end lives in namespace A, the other end lives in namespace B. They're permanently connected.

Veth pair connecting two namespaces
ip link add eth1 netns host1 type veth peer eth1b netns bridge1
ip link add
Create a new network link (interface).
eth1
Name of the first end of the cable. This end will be placed in the host1 namespace.
netns host1
Put the first end into the "host1" namespace. Like pushing one end of the cable through the wall into host1's apartment.
type veth
The type of link to create = virtual ethernet pair. This is what makes both ends exist and be connected.
peer
Keyword meaning "and here is the other end."
eth1b
Name of the second end of the cable. The "b" suffix is just convention — the script uses this to track which end is which.
netns bridge1
Put the second end into the "bridge1" namespace. This end will be plugged into the switch.
Why "bring up" an interface? Creating a veth pair doesn't automatically activate it. It's like plugging in a cable but the power switch is still off. You need to run ip link set eth1 up to turn it on. This is called "bringing the interface up."
Section 04

Linux Bridge — The Virtual Switch

A Linux bridge is software that behaves exactly like a physical network switch. It lives inside a namespace and all cables plugged into it get forwarded intelligently.
Analogy — The Building's Mailroom

Each apartment (namespace) is a separate unit. The bridge is the building's mailroom in the basement. Every apartment drops their outgoing mail here, and the mailroom worker (bridge device) figures out which apartment to deliver each letter to.

At first the mailroom doesn't know who lives where. So the first time PC-A sends a letter, the mailroom delivers it to every apartment (flooding). But it remembers "oh, PC-A lives in slot 3." Next time a letter arrives for PC-A, it goes directly to slot 3 — no more flooding to everyone.

This memory = MAC address table. This learning = bridge learning mode.

Linux bridge connecting 3 namespaces (simulating a switch)
ip link add br1 type bridge
ip link add
Create a new network device.
br1
Name of the bridge device. Just a name, like naming a file.
type bridge
Tell Linux this device should behave like a network switch. Linux will create a bridge device that learns MAC addresses and forwards frames.
ip link set eth1b master br1
eth1b
The veth end that's inside the bridge namespace. This is the "cable end" sticking out of the wall.
master br1
"Plug this cable into the switch." This attaches the interface as a port of the bridge. Like physically inserting a cable into a switch port.
Bridge vs. Hub A hub would forward every packet to every port, blindly. A bridge (and a real switch) is smarter — it learns which MAC address is on which port and delivers unicast frames only to the right port. However, broadcasts (ff:ff:ff:ff:ff:ff) are still forwarded to every port — that's unavoidable.

Script 1: create_bridge

Script 3: connect_bridges

This connects two bridge namespaces to each other — merging them into one larger broadcast domain.

connect_bridges — annotatedbash
connect_bridges() { local bridge1_nsname="$1" # Namespace of first bridge local bridge1_ifname="$2" # Bridge device name in first namespace local bridge2_nsname="$3" # Namespace of second bridge local bridge2_ifname="$4" # Bridge device name in second namespace # Generate names for the connecting veth pair local peer1_ifname="veth_${bridge2_ifname}" # e.g. "veth_br20" local peer2_ifname="veth_${bridge1_ifname}" # e.g. "veth_br10" # Create the veth pair connecting bridge1_ns ↔ bridge2_ns ip link add ${peer1_ifname} netns ${bridge1_nsname} type veth peer \ ${peer2_ifname} netns ${bridge2_nsname} # Turn both ends on ip netns exec ${bridge1_nsname} ip link set ${peer1_ifname} up ip netns exec ${bridge2_nsname} ip link set ${peer2_ifname} up # Plug each end into its respective bridge (like a patch cable between switches) ip netns exec ${bridge1_nsname} ip link set ${peer1_ifname} master ${bridge1_ifname} ip netns exec ${bridge2_nsname} ip link set ${peer2_ifname} master ${bridge2_ifname} }

The ethsend tool — Manual Ethernet Frames

This is a Python script that sends raw Ethernet frames. It bypasses TCP/IP completely and works purely at Layer 2 — like building and posting a letter by hand.

ethsend.py — fully annotatedpython
#!/usr/bin/env python3 # Usage: ethsend eth0 ff:ff:ff:ff:ff:ff 'Hello everybody!' # ethsend eth0 06:e5:f0:20:af:7a 'Hello specific MAC!' import fcntl # For reading interface info (MAC address) import socket # For creating the raw network socket import struct # For packing binary data import sys # For reading command-line arguments def send_frame(ifname, dstmac, eth_type, payload): # Create a RAW socket — bypasses TCP/IP, works at layer 2 # AF_PACKET = address family for raw packet access # SOCK_RAW = raw socket (no header stripping) s = socket.socket(socket.AF_PACKET, socket.SOCK_RAW) # Bind the socket to the network interface (e.g. "eth1") s.bind((ifname, 0)) # Get THIS interface's own MAC address using ioctl (kernel call) info = fcntl.ioctl(s.fileno(), 0x8927, struct.pack('256s', bytes(ifname, 'utf-8')[:15])) srcmac = ':'.join('%02x' % b for b in info[18:24]) # ↑ Extracts 6 bytes from position 18-24 of the ioctl response = MAC address payload_bytes = payload.encode('utf-8') assert len(payload_bytes) <= 1500 # Ethernet max payload is 1500 bytes (MTU) # BUILD the Ethernet frame manually: # [6 bytes dest MAC] + [6 bytes src MAC] + [2 bytes EtherType] + [payload] frame = human_mac_to_bytes(dstmac) + \ # Destination MAC human_mac_to_bytes(srcmac) + \ # Source MAC (ours) eth_type + \ # Protocol type (custom 0x7a05) payload_bytes # The actual message return s.send(frame) # Send it onto the wire! def human_mac_to_bytes(addr): return bytes.fromhex(addr.replace(':', '')) # "06:e5:f0:20:af:7a" → b'\x06\xe5\xf0\x20\xaf\x7a' (6 raw bytes) def main(): ifname = sys.argv[1] # e.g. "eth1" dstmac = sys.argv[2] # e.g. "ff:ff:ff:ff:ff:ff" (broadcast) payload = sys.argv[3] # e.g. "Hello all!" ethtype = b'\x7A\x05' # Custom EtherType — won't clash with real protocols send_frame(ifname, dstmac, ethtype, payload) if __name__ == "__main__": main()
Why AF_PACKET + SOCK_RAW? Normal programs use sockets at Layer 4 (TCP/UDP). AF_PACKET is a special socket type that lets you send/receive raw Ethernet frames — no IP, no TCP, just pure Layer 2. It requires root (or the CAP_NET_RAW capability) because it's very powerful.

Understanding tcpdump

tcpdump -i eth2 ether proto 0x7a05
tcpdump
Packet sniffer. It sits on a network interface and prints every packet that arrives, like a security camera for network traffic.
-i eth2
Which interface to listen on. The -i means "interface." Here we're watching eth2 — host2's network card.
ether proto 0x7a05
Filter: only show Ethernet frames with EtherType 0x7a05. This is the custom type ethsend uses. Without this filter, you'd see all kinds of background network traffic and it'd be messy. This keeps the output clean.
ip link show eth2 | grep ether | awk '{print $2}'
ip link show eth2
Show details about interface eth2. Output looks like:
3: eth2@eth2b: <BROADCAST> mtu 1500
    link/ether aa:bb:cc:dd:ee:ff brd ff:ff:ff:ff:ff:ff
| grep ether
Pipe the output, keep only the line containing "ether". The pipe | sends output of one command as input to the next. grep ether filters to just the line with the MAC address.
| awk '{print $2}'
Extract just the second word. awk splits each line into fields. $2 is the second field — which is the actual MAC address like aa:bb:cc:dd:ee:ff. We save that into ETH2_MAC.
What You Should Observe When host1 sends ff:ff:ff:ff:ff:ff, both host2 and host3's tcpdump windows print "Hello all!" — proving they're in the same broadcast domain. When host1 sends unicast to host2's MAC, only host2's tcpdump shows it.
Lab Example 02

Two Interconnected Switches

We create TWO separate bridges (switches) each with their own hosts — then connect the bridges together. The result is still ONE broadcast domain, because a bridge passes broadcasts.
Analogy — Two streets, connected by an alley

Street A has houses 10 and 11. Street B has houses 20 and 21. Initially they're separate — a shout on Street A doesn't reach Street B. But when you open an alley connecting them, a shout now carries across both streets. The alley = the veth pair connecting the two bridges.

The bridge passes broadcasts across — it has no wall. This is exactly the statement from the lab: "Multiple L2 segments can be bridged to create a bigger broadcast domain."

Lab Example 03

Hierarchical Internetworking

Real data centres don't use flat "everyone connects to everyone" topologies — they use a tree structure. Top-level switches connect to lower-level switches which connect to hosts. This lab shows that even a tree of switches is still one broadcast domain.
Analogy — A Tree of Mailrooms

Imagine two neighbourhood mailrooms (bridge100 and bridge200), each serving houses in their area. A central post office (bridge300) sits above both. Mail between the two neighbourhoods goes: house → local mailroom → central post office → local mailroom → house. A broadcast announcement goes all the way down every branch of the tree.

Topology — Example 3: Hierarchical switches
Lab 3 — Full command sequencebash
## ── SETUP: Create two lower-level segments ────────────────────── # Lower-layer segment I (Access Switch A) create_bridge bridge100 br100 create_end_host host100 eth100 bridge100 br100 create_end_host host101 eth101 bridge100 br100 # Lower-layer segment II (Access Switch B) create_bridge bridge200 br200 create_end_host host200 eth200 bridge200 br200 create_end_host host201 eth201 bridge200 br200 ## ── CREATE the higher-layer (core) switch ─────────────────────── create_bridge bridge300 br300 ## ── CONNECT both lower switches to the core switch ────────────── # This builds the tree topology: # br300 (core) # / \ # br100 br200 # / \ / \ # h100 h101 h200 h201 connect_bridges bridge100 br100 bridge300 br300 connect_bridges bridge200 br200 bridge300 br300 ## ── MONITOR on all non-source hosts ──────────────────────────── # Second host on bridge100 nsenter --net=/var/run/netns/host101 \ tcpdump -i eth101 ether proto 0x7a05 # Both hosts on bridge200 nsenter --net=/var/run/netns/host200 \ tcpdump -i eth200 ether proto 0x7a05 nsenter --net=/var/run/netns/host201 \ tcpdump -i eth201 ether proto 0x7a05 ## ── BROADCAST from host100 ────────────────────────────────────── # Path of broadcast: # host100 → br100 → bridge connection → br300 → bridge connection → br200 # → delivered to host200 AND host201 # Also: br100 delivers to host101 directly nsenter --net=/var/run/netns/host100 \ ethsend eth100 ff:ff:ff:ff:ff:ff 'Hello all!' ## ── CLEANUP ───────────────────────────────────────────────────── ip netns delete bridge100 bridge200 bridge300 ip netns delete host100 host101 host200 host201
The Key Takeaway from Lab 3 From every host's perspective, it doesn't matter whether the network is flat (all connected to one big switch) or hierarchical (tree of switches). A broadcast from any host reaches every other host. The logical view is always a single flat broadcast domain — no matter how complex the physical wiring is. This is exactly the diagram in the lab: "Logically, there is no difference between flat and hierarchical interconnections."

The Logical vs. Physical Insight — Summary No matter how you wire the switches — flat, tree, ring, star — if they're all connected via bridges with no routers in between, they form one broadcast domain. The hosts don't know or care about the physical layout. To split into multiple broadcast domains, you need either a router (physical boundary) or VLANs (logical boundary).
