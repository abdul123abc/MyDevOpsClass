Writing Kubernetes YAML
Without Memorising Anything

YAML in Kubernetes looks scary because of how much there is. But 90% of what you write follows the same 4-field skeleton. Learn the pattern once — and you never need to memorise again.

# The question everyone asks:
No, you don't memorise Kubernetes YAML. Nobody does — not even senior SREs at Google. You learn the structure and logic, then use tools (kubectl explain, VS Code extensions, --dry-run) to fill in the rest. This tutorial teaches you the pattern so everything clicks.
01 — Foundation

YAML Basics First

Kubernetes manifests are just YAML files. Before K8s-specific stuff, you need to understand how YAML itself works — it's simpler than it looks.

Rule 1 — Indentation Is Everything

YAML uses spaces (NOT tabs) to show nesting. Parent → child is shown by indenting the child by 2 spaces. Get this wrong and your manifest breaks. It's the #1 source of confusion.

spec:
Level 0
containers:
Level 1 — child of spec
- name: my-app
Level 2 — list item
image: nginx:1.25
Level 3 — child of list item
⚠ CRITICAL RULE

Always use 2 spaces per indent level. Never use Tab. Most editors let you configure Tab to insert 2 spaces — do that immediately. Mixing tabs and spaces will silently break your YAML.

Rule 2 — Data Types

strings.yaml
name: my-app          # plain string
image: "nginx:1.25"    # quoted string
message: |             # multi-line string
  line one
  line two
numbers-bools.yaml
replicas: 3          # integer
port: 8080           # integer
enabled: true        # boolean
readOnly: false      # boolean

Rule 3 — Lists vs Maps

list.yaml — starts with dash
ports:
  - 80
  - 443
  - 8080

containers:       # list of objects
  - name: app
    image: nginx
  - name: sidecar
    image: fluentd
map.yaml — key: value pairs
labels:             # map = nested key:value
  app: my-app
  env: production
  version: v2

annotations:        # also a map
  team: backend
  owner: abdullah
💡 TIP — Quoted strings

If a string contains :, {, }, [, ], or starts with a number, wrap it in quotes. Example: version: "1.25" — otherwise YAML might misinterpret it.

02 — The Universal Skeleton

Every Manifest Has These 4 Fields

No matter what Kubernetes object you're creating — Pod, Deployment, Service, Job, whatever — the top-level structure is always the same 4 fields. Learn this once and you know the skeleton of every K8s YAML ever written.

universal-skeleton.yamlhover the underlined parts ↓
apiVersion: apps/v1
  
📌 API Group / Version
Tells K8s which API to use. Different objects live in different API groups. apps/v1 is for Deployments/DaemonSets. v1 (no prefix) is for Pods/Services. batch/v1 is for Jobs. You look this up — nobody memorises it.


kind: Deployment
  
📌 Object Type
What you're creating. Deployment, Pod, Service, ConfigMap, Secret, Job, CronJob, Ingress, PersistentVolumeClaim, Namespace, etc. Capital letter always.


metadata:
  name: my-app
  namespace: production
  labels:
    app: my-app
  
📌 Metadata
Identity of the object. name is required — it's how kubectl and other resources refer to this object. namespace is optional (defaults to "default"). labels are key-value tags used for selection — VERY important.


spec:
    # This is where everything interesting lives.
    # Contents depend on the 'kind' above.
    # A Pod spec is different from a Service spec.
  
📌 Specification
The desired state you want Kubernetes to achieve. This is the bulk of the YAML and varies by kind. THIS is the part that looks different between object types. Everything else above is always the same.
Field Required? What it does
apiVersion required
Which API group/version handles this object. Common ones: v1 · apps/v1 · batch/v1 · networking.k8s.io/v1
kind required
The object type. Pod · Deployment · Service · Job · ConfigMap · Secret · Ingress · PersistentVolumeClaim
metadata.name required
Unique name for this object within its namespace. Lowercase, hyphens OK, no spaces. e.g. user-service · mysql-statefulset
metadata.namespace optional
Which namespace to put this in. Defaults to default if omitted. You know namespaces are like office floors — use them to separate environments.
metadata.labels optional*
Key-value tags. Technically optional but practically required — Services use labels to find which Pods to route to. Always add app: your-app-name at minimum.
spec required
The desired state. Contents depend on kind. This is the part that varies — and the part kubectl explain helps you with.
💡 KEY INSIGHT

The first 3 fields (apiVersion, kind, metadata) are always identical in structure regardless of what you're creating. Only spec changes between object types. Once you know the spec for a Deployment, you know 75% of any Deployment YAML you'll ever write.

03 — Real Examples, Fully Annotated

The 6 Objects You'll Write Most

Click each tab. Every field is explained so you understand what it does — not just what to type.

A Pod is the smallest unit in K8s — one or more containers running together. You rarely create Pods directly (Deployments do it for you), but understanding Pod spec is essential because Deployment/StatefulSet/Job spec all contain a Pod template inside them.

pod.yamlhover underlined fields
apiVersion: v1                     # Pods are in the core v1 group
kind: Pod
metadata:
  name: my-pod
  labels:
    app: my-app                    # Services find this pod using this label
spec:
  containers:                    # LIST of containers in this pod
  
📌 containers is a list (notice the dash below). You can have multiple containers per pod — main app + sidecar. Each gets the same IP address and can talk to each other on localhost.

  - name: app-container          # name of this specific container
    image: nginx:1.25
    
📌 image — Docker image to run. Format: name:tag. Always pin a specific tag (1.25) instead of :latest in production. :latest is non-deterministic — different machines may pull different versions.

    ports:
    - containerPort: 80          # port the app listens on INSIDE container
    
📌 containerPort is purely documentation — it doesn't actually open the port. The port is already open because the app inside listens on it. This just tells humans/tools which port to expect.

    resources:
      requests:                    # minimum guaranteed resources
        cpu: "250m"               # 250 millicores = 0.25 CPU core
        memory: "128Mi"           # 128 mebibytes RAM
      limits:                      # maximum allowed resources
        cpu: "500m"
        memory: "256Mi"           # exceed this = OOMKilled
    
📌 resources — requests is what the scheduler uses to place the pod (VPA right-sizes this). limits is the hard ceiling. Exceed memory limit → pod is killed (OOMKilled). Exceed CPU limit → pod is throttled (not killed). Always set both.

    env:                             # environment variables
    - name: DB_HOST
      value: "mysql-service"       # hardcoded value
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:              # pull from a Secret instead
          name: db-secret
          key: password

This is what you use for stateless apps in production. A Deployment manages a set of identical Pod replicas and handles rollouts/rollbacks. It wraps a Pod template inside its spec.

deployment.yaml
apiVersion: apps/v1              # Deployments are in apps group
kind: Deployment
metadata:
  name: user-service
  namespace: production
  labels:
    app: user-service
spec:
  replicas: 3
  
📌 replicas — how many identical Pod copies to run. HPA will override this automatically when it scales. Start with a small number like 2-3.

  selector:
    matchLabels:
      app: user-service
  
📌 selector — tells the Deployment which Pods it owns. Must match the labels in the template below. If these don't match, the Deployment won't know which pods are "its" pods. This is the most common source of bugs for beginners.

  template:                        # ← THIS IS THE POD TEMPLATE
    metadata:
      labels:
        app: user-service         # must match selector.matchLabels above
    spec:                          # ← same as Pod spec from here on
      containers:
      - name: user-service
        image: my-registry/user-service:v2.1
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: "250m"
            memory: "256Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"
  
📌 template — the Pod blueprint. Every pod the Deployment creates will look like this. Notice it has its own metadata and spec — it IS a Pod definition, just nested inside the Deployment.

  strategy:
    type: RollingUpdate          # update pods gradually, not all at once
    rollingUpdate:
      maxUnavailable: 1          # max pods down at any time during update
      maxSurge: 1               # max extra pods during update

You know Services deeply from the previous tutorial. Here's how to write them. The key field is selector — it must match the pod labels exactly.

service.yaml
apiVersion: v1                   # Services are in core v1 group
kind: Service
metadata:
  name: user-service
  namespace: production
spec:
  type: ClusterIP                # ClusterIP | NodePort | LoadBalancer
  
📌 type — ClusterIP (internal DIP, default), NodePort (hole on every node), LoadBalancer (VIP via cloud). You know all of these from the Services tutorial.

  selector:
    app: user-service           # routes to pods with THIS label
  
📌 selector — this is how the Service finds its pods. It looks for ALL running pods that have matching labels and automatically routes traffic to them. Pod comes up = added to pool. Pod dies = removed. This is the magic.

  ports:
  - port: 80
    
📌 port — the port the Service listens on. This is what callers use. "Call user-service on port 80."

    targetPort: 8080
    
📌 targetPort — the port the pod actually listens on. Traffic hits Service:80 → forwarded to Pod:8080. These can be different — common pattern is expose on 80, app runs on 8080.

    protocol: TCP               # TCP is default, can be UDP

---                               # Three dashes = new document in same file
# LoadBalancer example:
apiVersion: v1
kind: Service
metadata:
  name: api-gateway
spec:
  type: LoadBalancer             # provisions cloud LB = your VIP
  selector:
    app: api-gateway
  ports:
  - port: 443
    targetPort: 8443

ConfigMaps store non-sensitive configuration that you want to inject into pods — database hostnames, feature flags, config files. Keeps config out of your Docker image.

configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:                              # ConfigMap uses 'data' not 'spec'
  DB_HOST: "mysql-service"
  DB_PORT: "3306"                 # numbers are strings here
  APP_ENV: "production"
  MAX_CONNECTIONS: "100"
  config.json: |                   # can store entire files!
    {
      "logLevel": "info",
      "timeout": 30
    }

📌 data — ConfigMap uses data instead of spec. Each key becomes an environment variable or a file, depending on how you mount it in the pod.


---
# Using ConfigMap in a Pod:
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
  - name: app
    image: my-app:v1
        # Method 1: inject all keys as env vars
    envFrom:
    - configMapRef:
        name: app-config
        # Method 2: mount config.json as a file
    volumeMounts:
    - name: config-vol
      mountPath: /etc/config      # accessible at /etc/config/config.json
  volumes:
  - name: config-vol
    configMap:
      name: app-config

Secrets are like ConfigMaps but for sensitive data — passwords, tokens, TLS certs. Values must be base64 encoded (not encrypted — just encoded). For real security, use Vault or Sealed Secrets on top.

secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque                     # Opaque = generic secret

📌 type — Opaque means "generic, I'll define the structure." Other types: kubernetes.io/tls (for TLS certs), kubernetes.io/dockerconfigjson (for registry auth). Most of the time you'll use Opaque.

data:                              # values MUST be base64 encoded
  password: cGFzc3dvcmQxMjM=       # base64 of "password123"
  username: YWJkdWxsYWg=          # base64 of "abdullah"

📌 Values in Secret.data must be base64 encoded. To encode: echo -n "mypassword" | base64. To decode: echo "bXlwYXNzd29yZA==" | base64 -d. Use stringData instead to write plain text — K8s will encode it for you.


---
# OR use stringData to skip manual base64:
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
stringData:                        # plain text — K8s encodes it
  password: "password123"         # ← much easier
  username: "abdullah"

---
# Pulling from Secret in a Pod:
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-secret              # the Secret's metadata.name
      key: password               # the key inside the Secret

PVC = your storage booking request. You know the PV/PVC/StorageClass model from the persistence tutorial. Here's how you write it.

pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mysql-storage
spec:
  accessModes:
  - ReadWriteOnce               # RWO: one node. RWX: many nodes.
  
📌 ReadWriteOnce (RWO) — only one node can mount this as read-write. That's fine for a single database. ReadWriteMany (RWX) needed if multiple pods on different nodes need to write — requires NFS or EFS.

  storageClassName: fast-ssd     # which StorageClass to use
  
📌 storageClassName matches a StorageClass that your cluster has. It tells K8s "create storage of this type." Your cloud provider usually sets up default classes — run kubectl get storageclasses to see what's available.

  resources:
    requests:
      storage: 10Gi               # I need at least 10 gibibytes
      
📌 10Gi = 10 gibibytes (1024-based). 10G = 10 gigabytes (1000-based). Use Gi for K8s to be precise. K8s will find a PV with storage ≥ 10Gi and bind it.


---
# Mounting PVC in a Pod:
apiVersion: v1
kind: Pod
metadata:
  name: mysql
spec:
  containers:
  - name: mysql
    image: mysql:8.0
    volumeMounts:
    - name: data
      mountPath: /var/lib/mysql   # where in container to mount
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: mysql-storage    # reference the PVC by name
04 — Common Mistakes

Gotchas That Will Catch You

These mistakes are nearly universal when people start writing K8s YAML. Know them now and save yourself hours of debugging.

🎯

Selector ↔ Label mismatch (most common bug)

A Deployment's selector.matchLabels must exactly match the template.metadata.labels. A Service's selector must exactly match pod labels. One typo = service routes to nothing, pods get no traffic. Always cross-check these.

📐

Wrong indentation level

The dash in a list item (-) and the properties under it must align correctly. Properties of a list item are indented 2 spaces from the dash. Missing or extra spaces silently break the structure.

🔢

Forgetting quotes on numeric strings

If a field expects a string but you write a number, YAML parses it as an integer and K8s rejects it. Example: port: "3306" — if the field expects a string, quote it. Use kubectl explain to check the expected type.

📦

Using :latest image tag in production

Always pin your image tag: image: my-app:v2.1.3. Using :latest means different nodes can pull different versions, rollbacks become unpredictable, and pods restart with unexpected code changes.

🔑

Forgetting namespace context

A Service in namespace production can only be reached from other resources in production by short name. Cross-namespace calls need the full DNS: service-name.namespace.svc.cluster.local.

💾

Mixing tabs and spaces

YAML strictly forbids tabs for indentation. Your editor may silently insert tabs. Configure your editor to use 2 spaces when Tab is pressed. VSCode: Settings → "Editor: Tab Size" = 2 and "Editor: Insert Spaces" = true.

🔐

Putting secrets in ConfigMaps

ConfigMaps are not encrypted. Never put passwords, tokens, or private keys in a ConfigMap. Always use kind: Secret. Even better, use external secret management (HashiCorp Vault, AWS Secrets Manager).

05 — You Don't Need To Memorise

Tools That Do The Work For You

Here's the real answer to your question. Real engineers use these daily — not their memory.

kubectl explain — Your Built-in Docs

This is your most powerful tool. It shows you every field of every object, their types, and descriptions — right in the terminal. You never need to google what a field is called.

$kubectl explain deployment KIND: Deployment / VERSION: apps/v1 / DESCRIPTION: A Deployment provides declarative updates for Pods...
$kubectl explain deployment.spec FIELDS: replicas <integer> · selector <Object> -required- · template <Object> -required- · strategy <Object>...
$kubectl explain deployment.spec.template.spec.containers FIELDS: name <string> -required- · image <string> · ports <[]Object> · resources <Object> · env <[]Object>...
$kubectl explain pod.spec.containers.resources --recursive Shows entire tree of fields under resources

--dry-run + -o yaml — Generate YAML Automatically

Instead of writing YAML from scratch, let kubectl generate the skeleton and edit it:

$kubectl create deployment my-app --image=nginx:1.25 --replicas=3 --dry-run=client -o yaml Outputs complete Deployment YAML → redirect to file and edit
$kubectl create deployment my-app --image=nginx --dry-run=client -o yaml > my-deployment.yaml Saves the generated YAML to a file — then edit as needed
$kubectl create service clusterip my-svc --tcp=80:8080 --dry-run=client -o yaml Generates a ClusterIP Service YAML

More Essential Tools

VSCode + YAML Extension

Install "YAML" by RedHat. Gives you autocomplete, inline docs, and error highlighting directly in your editor as you type.

kubectl apply --dry-run

Validate your YAML against the live API server without actually creating anything. Catches typos and schema errors.

kubectl diff -f

Shows what would change if you applied a file — like git diff but for live K8s state vs your YAML.

kubeval / kubeconform

CLI tools that validate YAML files offline against K8s schemas. Good for CI/CD pipelines.

Lens IDE

GUI for Kubernetes. Shows live state of all objects, lets you edit YAML in-browser, great for visualising what's running.

kubectl get -o yaml

Get the YAML of any running object: kubectl get deployment my-app -o yaml. Copy an existing working manifest and modify it.

✅ THE REAL WORKFLOW

1. Generate a skeleton: kubectl create ... --dry-run=client -o yaml > file.yaml
2. Open in VSCode with YAML extension — autocomplete fills in what you need
3. Validate: kubectl apply --dry-run=server -f file.yaml
4. Apply: kubectl apply -f file.yaml
5. Debug: kubectl describe and kubectl logs
