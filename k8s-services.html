01 — The Problem

Why Do Services Even Exist?

Before we touch any Service type, we need to feel the pain they solve. Because Kubernetes Services are not some arbitrary abstraction — they exist because pods are fundamentally broken for direct communication.

Your cluster is a large corporate office city. Pods are employees. The problem? Employees move desks constantly — they get fired, rehired, promoted to different buildings. Every time they move, they get a new phone number. If you've been calling John directly on his personal phone, and John moves to a new desk tomorrow with a new number, your call fails.

A Service is the company's main office line — it stays the same no matter how many times employees come and go behind it.

Here is the cold technical truth:

❌ Without Services

  • Pod gets IP 10.0.1.5
  • Pod crashes, restarts
  • New pod gets IP 10.0.1.9
  • All hardcoded callers → broken ☠️
  • 10 replicas = 10 IPs to track manually

✅ With Services

  • Service gets stable IP 10.96.0.1
  • Pods come and go freely
  • Callers always reach 10.96.0.1
  • Service routes to any healthy pod
  • 10 replicas = 1 IP, auto load-balanced
02 — Foundation

The Flat Network You Already Know

You mapped this perfectly in our earlier chat — and it's exactly right:

KubernetesAWS / Networking EquivalentWhy It Matches
ClusterVPC (Virtual Private Cloud)Isolated private network space with its own CIDR block
NodeSubnetEach node owns a sub-range of IPs (e.g. 10.0.1.0/24)
PodDevice / EC2 instanceGets its own unique IP from the node's range
CNI PluginVPC Route TablesPrograms routes so pods on different nodes can reach each other

The Golden Rule: Every pod can reach every other pod in the cluster directly by IP address — same node or different node — without NAT, without port mapping. This is the Kubernetes flat network guarantee, enforced by the CNI plugin.

But here's the catch — pod IPs are ephemeral. They change every restart. This is the exact problem Services solve. Let's go through each type.

03 — Service Type 1

ClusterIP — The Internal DIP

You nailed this one: ClusterIP is the DIP (Destination IP) in a load balancer setup. It's a stable virtual IP that lives only inside the cluster, backed by iptables/IPVS rules that distribute traffic across all healthy pods behind it. Just like a DIP in an F5 setup — the VIP in front sends to the DIP, the DIP distributes to real servers (pods).

ClusterIP is the default Service type. When you create any Service without specifying a type, you get a ClusterIP. It's the foundation everything else builds on.

🔑 The Trick: The ClusterIP address is never actually reached by any packet. When a pod sends traffic to 10.96.0.1:80, the Linux kernel's iptables rules (written by kube-proxy) intercept the packet and rewrite the destination to one of the real pod IPs — before it ever leaves the node. The ClusterIP is a virtual illusion enforced in the kernel.

apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  type: ClusterIP        # default — can omit this line
  selector:
    app: user-app        # routes to pods with this label
  ports:
  - port: 80             # port the Service listens on
    targetPort: 8080     # port the pod actually runs on

Use when: Microservice-to-microservice communication inside the cluster. Your auth-service calling your user-service. Nobody outside the cluster needs to be involved.

04 — Service Type 2

NodePort — The Side Door on Every Floor

The VIP (LoadBalancer) doesn't exist yet. You need external traffic to reach your app. So you tell every security guard at every building entrance (every Node): "If anyone shows up and asks for door #30080, walk them to the reception desk (ClusterIP) inside."

NodePort punches a hole through the firewall on every single node simultaneously at a specific high-numbered port (30000–32767). Anyone who knows any node's IP + that port number can get in.

Use when: Local development, bare-metal clusters without a cloud LB, quick demos. Not recommended for production — it exposes high ports on every node and forces clients to know node IPs directly.

05 — Service Type 3

LoadBalancer — The VIP

You also nailed this: LoadBalancer = VIP. It's the single clean public-facing IP the world knows about. A professional front-of-building reception desk with its own well-known address on Google Maps. Visitors just go to that one address — they never need to know about individual nodes or ports.

LoadBalancer builds on top of NodePort and ClusterIP. Kubernetes calls your cloud provider's API (AWS, GCP, Azure) to provision an actual load balancer with a real public IP. The full chain is:

The Russian Nesting Dolls 🪆

LoadBalancer, NodePort, and ClusterIP are not separate things — they nest inside each other. Creating a LoadBalancer Service silently creates a NodePort and a ClusterIP underneath.

LoadBalancer ← outermost layer (VIP)
NodePort ← middle layer (hole on every node)
ClusterIP ← innermost layer (the DIP, always present)

Use when: Any production service exposed to the internet. Your main API, your frontend, anything users outside the cluster need to reach.

06 — Service Type 4

ExternalName — The Company DNS Alias

You compared it to /etc/hosts — the spirit is exactly right. But it's more precise: ExternalName is like adding a record to your company's internal DNS server (not a local hosts file) saying: "whenever anyone inside asks for my-db, redirect them to prod.database.amazonaws.com instead." It's cluster-wide, not machine-specific. The underlying mechanism is a DNS CNAME record, not an IP mapping.

apiVersion: v1
kind: Service
metadata:
  name: my-db
spec:
  type: ExternalName
  externalName: prod.database.amazonaws.com   # the external DNS name

Use when: You're migrating a service (like an RDS database) into the cluster and want your app code to keep calling the same name. Or you want to abstract a third-party API behind a clean internal name so your code never changes even if the backend URL does.

07 — Special Variant

Headless Service — The Staff Directory

Normal ClusterIP = receptionist who picks one random person to handle your call.

Headless = you get handed the full staff directory for the department. You decide exactly who to call. This is critical for StatefulSets (MySQL, Kafka, Cassandra) where the client needs to reach Pod 0 specifically, not just any random replica.

Created by setting clusterIP: None. Returns DNS A records for all pod IPs directly instead of a single virtual IP. Each StatefulSet pod gets its own stable DNS name:

mysql-0.mysql-svc.namespace.svc.cluster.local  →  10.0.1.2
mysql-1.mysql-svc.namespace.svc.cluster.local  →  10.0.1.3
mysql-2.mysql-svc.namespace.svc.cluster.local  →  10.0.2.5

When you have a MySQL primary + replicas, your app connects directly to mysql-0 for writes (primary) and distributes reads to mysql-1, mysql-2. A normal ClusterIP can't give you that precision — it would randomize traffic across all three, sending writes to replicas (disaster).

08 — The Engine Room

kube-proxy — The Road Sign Painter

kube-proxy is not a load balancer in the traditional sense. It doesn't sit in the traffic path like Nginx or HAProxy. Think of it as a traffic warden who paints road signs — it never stands in the road directing cars itself, but it ensures all the signs (iptables rules) are correct so the road system routes cars automatically. The actual traffic never touches kube-proxy — it flows kernel → iptables → pod directly.

Traditional Load Balancer

Sits in the traffic path
Processes every packet
Has its own IP address
Nginx, HAProxy, Envoy

kube-proxy

Never touches traffic
Only updates iptables/IPVS rules
No traffic IP of its own
Rule programmer / control agent

kube-proxy runs as a DaemonSet — one instance on every node (you already know DaemonSets). It watches the Kubernetes API for Service and Endpoint changes. When a pod dies and a new one starts (new IP), kube-proxy updates the iptables rules on every node within seconds so traffic is re-routed automatically.

09 — Layer 7 Routing

Ingress — The Smart Hotel Receptionist

Without Ingress, exposing 5 microservices means 5 LoadBalancer Services → 5 cloud load balancers → 5 public IPs → 5 separate bills.

Ingress is like a smart hotel receptionist sitting at one front desk with one address. You walk in and say "I'm here for the restaurant" or "I'm here for the spa." They read your request and route you to the right place. One entrance. Many destinations.

⚡ Key Distinction: Everything below Ingress (LoadBalancer, NodePort, ClusterIP) operates at Layer 4 — they understand IPs and ports only. Ingress operates at Layer 7 — it understands HTTP hostnames, URL paths, headers. That's its superpower.

The Two Parts You Must Know

📋

Ingress Resource

A YAML config file — just routing rules. Declares "traffic to /api goes to user-service." It does absolutely nothing by itself — it's just a document.

The Policy Document
⚙️

Ingress Controller

The actual running software that reads and enforces those rules. Kubernetes does NOT ship one — you install it. Popular: Nginx, Traefik, AWS ALB, Kong.

The Enforcer

One domain, split by URL path. Same hostname, different services based on what path the request hits:

Multiple subdomains, each routed to a different service. The Ingress Controller reads the Host HTTP header:

api.myapp.com      ──→  user-service:80
shop.myapp.com     ──→  product-service:80
admin.myapp.com    ──→  admin-service:80

All three subdomains point to the same LoadBalancer IP in DNS. The Ingress Controller inspects the Host header and routes accordingly. Zero extra cloud LBs needed.

Ingress handles HTTPS encryption at the edge. Traffic inside the cluster can stay as plain HTTP — the TLS is terminated at the Ingress Controller.

Like a security checkpoint at the building entrance that verifies your encrypted ID card, then lets you walk freely inside without showing it again at every desk.

spec:
  tls:
  - hosts:
    - myapp.com
    secretName: myapp-tls-cert   # TLS cert stored as K8s Secret
  rules:
  - host: myapp.com
    http:
      paths:
      - path: /
        backend:
          service:
            name: my-service
            port: 80

Annotations are controller-specific configuration sticky notes on the Ingress resource. The main rules say where to route; annotations control how to route:

annotations:
  nginx.ingress.kubernetes.io/rate-limit: "100"       # 100 req/min
  nginx.ingress.kubernetes.io/rewrite-target: /        # URL rewrite
  nginx.ingress.kubernetes.io/ssl-redirect: "true"     # force HTTPS
  nginx.ingress.kubernetes.io/proxy-body-size: "50m"   # upload limit
  nginx.ingress.kubernetes.io/enable-cors: "true"      # CORS headers

These are Nginx-specific. AWS ALB, Traefik, and Kong all have their own annotation schemas. This is why the Ingress spec says "use this controller" — annotation keys are tied to the controller implementation.

11 — Decision Guide

Which Service Do I Use?

🔒

Internal communication?

Pods calling other pods within the cluster — auth calling users, frontend calling backend.

Use ClusterIP
🛠️

Testing locally / bare metal?

No cloud provider. You want external access without setting up a full LB.

Use NodePort
🌐

Production internet traffic?

Real users on the internet hitting your API or frontend. Cloud provider available.

Use LoadBalancer
🗺️

Multiple services, 1 domain?

You want /api, /shop, /admin to route to different services. One public IP.

Use Ingress
🔗

Wrapping external service?

RDS database, third-party API — want a clean internal name for it.

Use ExternalName
🗄️

Stateful app (DB, Kafka)?

Need to talk to specific named pods, not random replicas. StatefulSet use case.

Use Headless
